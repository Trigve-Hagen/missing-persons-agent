Quick start: Go to the Person click the Edit button and then click the Save person to database. Click Set State button on the Person page to set the person as state. Download Ollama from Ollama. Make sure its on. Go to Models ad set both name and Model to phi3:mini and click the Save button. Click Set State button on the Models page to set the person as state. Got to the Chat Box page, type Hello where it says Type a message... and click the Send button.

Gather information by filling out the Person, Phones, Emails, Addresses, Aliases and uploading Pdfs with information on the case. I will add Excels and Word docs in the future. You can also use APIs and Rss feeds to find data although this is not finished yet. Once you have uploaded the data use agents, promts and Qestions to find connections.

The models at this time are Ollama models. You will need to find models trained on missing person investigating or find information on how to investigate a missing person and train the model on how to get it done.

By training the model I mean uploading documents and save them to the vector database. RAGs are unique models that are pre-trained but have the ability to use data saved in the vector database to continue learning on. You can create your own way of doing things based upon your own ideas by creating a pdf with your own strategy.

There is a model on the Notice page that gives 10 sugestions on anything you ask it for. By providing really precise prompts and questions you can get really detailed answers. Study prompt engineering.

Files and Person have a save to vector database button that you will need to click to save the data to the Chroma vector database. The person consolidates all the data in the list of entites under the Person menu into one chunk per person. The files saves separately.

Chunking

When chunking, do periods or commas matter when separating expressions like 'Attribute: value' and is that experssion the best way to add mutiple expressions to a chunk?

Yes. Periods and commas matter when chunking text for an LLM, primarily because they dictate semantic flow and structural coherence. The expression 'Attribute: value' is a highly effective way to add metadata, but it requires specific handling in your chunking strategy.

Do Periods and Commas Matter?
  • Periods: Yes. During recursive character splitting, splitters often use periods as natural separators. If you arbitrarily truncate chunks without respecting sentence-ending punctuation, you risk cutting off thoughts midway and confusing the LLM.
  • Commas: Yes, but more for the language model's comprehension than the raw chunking logic. Commas delineate items in a list or within an 'Attribute: value' expression, ensuring the model parses them as distinct points.
Is 'Attribute: value' the Best Strategy?
Using key-value pairs ('Attribute: value') is an excellent strategy for adding context, but you should not append a raw list of them at the end of a long, otherwise unrelated text chunk. To get the most out of this approach, apply it properly using metadata injection:
  1. Metadata Prepending (Recommended)
    Instead of dumping attributes into the main text, prepend the key-value expressions to the beginning of your text chunk. This allows the LLM and embedding model to absorb the context immediately before reading the content.
    • Example: [Category: Technical] [Doc_Section: API Reference] [Author: Team A] The following function is used to...
  2. Document Metadata (The Cleanest Approach)
    Most vector database tools (like LangChain or LlamaIndex) support dedicated metadata dictionaries. Instead of writing 'Author: John Doe' in the text itself, you can store {"Author": "John Doe"} as a property of the chunk.
Why Metadata Dictionaries Are Better Than Raw Text
  • Cleaner Embeddings: It prevents your embedding model from getting confused by literal text attributes mixed into the document's narrative.
  • Filterable Search: It allows you to filter search results by exact metadata before even passing the chunk to the LLM (e.g., "Only search chunks where Document_Type = Policy").
  • Saves Tokens: It prevents you from wasting expensive context window tokens on long lists of attributes.
Best Practices to Remember
  1. Never break mid-expression: If you are building chunks manually, write a custom script that ensures 'Attribute: value' groups are kept together entirely within a single chunk, rather than split across two.
  2. Limit to 3-5 key expressions: Overloading your chunks with too many attributes can dilute the importance of the actual content.
  3. Use sentence boundaries: If you use punctuation-aware chunking, make sure your code accounts for abbreviations or decimals so the splitter doesn't mistakenly create a new chunk mid-sentence

Ensure the text fits your language model's context window. What does this mean?

When a text "fits your language model's context window," it means the total length of the text you provide does not exceed the maximum amount of information the AI can process at one time.

Think of the context window as the AI's "working memory" or a highly restricted desktop space.

Here is exactly what that means and why it matters:

  • What it includes: The context window encompasses both the prompt you type (the input) and the response the AI generates (the output).
  • Token limits: AI models process text in chunks called "tokens" (roughly equal to 4 characters, or about 0.75 words). Context windows range from a few thousand tokens for older models to millions of tokens in newer ones.
  • What happens if it's too big: If your text exceeds the limit, the AI will reject the prompt entirely, crash, or ask you to shorten it.
  • What happens if it's near the limit: Even if the text fits, pushing the absolute maximum limit can sometimes cause the AI to "forget" details from the beginning of the text or lose focus during generation.

How to fix it: If a document is too large, you can summarize it in chunks, share only the most relevant paragraphs, or use external file-reading tools if your platform supports them.
